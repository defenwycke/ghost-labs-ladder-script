PRESET COUNTER BOARD VOTE

Multi-party board vote with preset counter. The counter tracks approvals; when the preset threshold is reached, the funds are released. HYSTERESIS_VALUE limits payout size. Demonstrates governance-controlled spending with programmable thresholds.

CTR_PRE SIG HYST_VAL REC_MOD MULTISIG SCHNORR · SCHEME
Rung Structure

Three rungs: VOTE (SIG + CTR_PRE + HYST_VAL + REC_MOD for recursive voting), RELEASE (SIG when counter reaches preset), VETO (dual SIG for override).

Each row is a rung (spending path). Rungs are OR — satisfy any one rung to spend. Blocks within a rung are joined by + meaning AND — all must pass.

How Preset Counter Board Vote Works
1.
The VOTE rung uses COUNTER_PRESET to track board approvals. Each vote increments current toward preset. HYSTERESIS_VALUE constrains payout size within a band.
2.
RECURSE_MODIFIED mutates the counter on each vote, creating a carry-forward with the incremented count. The same conditions persist but the vote count advances.
3.
When current = preset, the COUNTER_PRESET block flips behavior and the RELEASE rung becomes spendable. The board has reached quorum.
4.
The VETO rung requires two signatures — a designated pair can override and cancel the vote at any point, providing a safety mechanism.
