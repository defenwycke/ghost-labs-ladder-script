VAULT WITH UNVAULT + CLAWBACK

Bitcoin vault pattern using the VAULT_LOCK block type. A v4 RUNG_TX creates a 2-path output: the hot key can spend after a CSV relative delay (unvault path), while the recovery key can sweep immediately (clawback path). The delay gives the owner time to detect and claw back unauthorized spends.

VAULT SIG SCHNORR · SCHEME
Rung Structure

Two rungs encode the spending paths. UNVAULT uses the hot key with a CSV delay. CLAWBACK uses the recovery key with no delay &mdash; immediate recovery.

Each row is a rung (spending path). Rungs are OR — satisfy any one rung to spend. Blocks within a rung are joined by + meaning AND — all must pass.

How Vault With Unvault + Clawback Works
1.
The VAULT_LOCK block is self-contained: it commits both the hot key and recovery key PUBKEY_COMMITs, plus a numeric delay (CSV blocks). One block encodes both spend paths.
2.
Evaluator hot-key path: verify hot key signature, enforce nSequence ≥ delay (BIP-68 relative timelock). This gives the owner a time window to detect unauthorized access.
3.
Evaluator recovery path: verify recovery key signature, no timelock required. The cold key can sweep immediately, acting as a clawback if the hot key is compromised.
4.
The vault spends by mining enough blocks to satisfy the CSV delay, then signing with the hot key. In production, the watchtower monitors for unauthorized unvault attempts during the delay window.
