TRANSACTION EXAMPLE · COVENANT

Dead Man's Switch (Inheritance)

Bitcoin inheritance pattern using RECURSE_SAME + LATCH_SET. The owner must periodically sign a keepalive transaction to maintain control. Each spend re-encumbers the output with identical conditions via RECURSE_SAME, resetting the CSV clock. If the owner fails to sign within the timeout window, the heir can claim the funds via the INHERIT path.

RECURSE_SAME · TYPE 0x0D LATCH_SET · TYPE 0x1D SCHNORR · SCHEME 0x01
Rung Structure

The funding transaction creates two outputs, each with a single rung (spending path). Vout 0 is the owner's keepalive path — a recursive covenant that re-encumbers itself. Vout 1 is the heir's claim path, gated by a CSV relative timelock.

The ALIVE rung combines three block types: SIG proves the owner is alive, LATCH_SET provides state tracking (precondition: state must be 0), and RECURSE_SAME enforces that the spending output carries byte-identical conditions — the covenant loop. How Dead Man's Switch Works
1.
The funding transaction creates two outputs. Vout 0 (ALIVE) contains a single rung with three blocks: SIG (owner's PUBKEY_COMMIT), LATCH_SET (state=0), and RECURSE_SAME (max_depth=10). Vout 1 (INHERIT) contains CSV (26,000 blocks ~6 months) + SIG (heir's key).
2.
To prove they are alive, the owner spends vout 0 with a valid signature. The evaluator checks: SIG is valid, LATCH_SET precondition passes (state==0), and the spending output carries byte-identical conditions (RECURSE_SAME). The output is re-encumbered with the same covenant.
3.
Each keepalive resets the relative timeout on the INHERIT output. As long as the owner keeps spending within the CSV window, the heir cannot claim. The covenant can loop up to max_depth times (10 in this example).
4.
If the owner fails to sign within 26,000 blocks (~6 months), the heir satisfies vout 1 by waiting out the CSV lock and providing their own signature. The INHERIT path has no recursion constraint — it spends freely to any destination.
