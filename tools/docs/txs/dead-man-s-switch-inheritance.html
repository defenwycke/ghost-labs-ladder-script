DEAD MAN'S SWITCH (INHERITANCE)

Bitcoin inheritance pattern using RECURSE_SAME + LATCH_SET. The owner must periodically sign a keepalive transaction to maintain control. Each spend re-encumbers the output with identical conditions via RECURSE_SAME, resetting the CSV clock. If the owner fails to sign within the timeout window, the heir can claim via the INHERIT path.

SIG LATCH_S REC_SAME CSV SCHNORR · SCHEME
Rung Structure

Two outputs with separate paths. ALIVE: owner keepalive with recursive covenant (SIG + LATCH_SET + RECURSE_SAME). INHERIT: heir claim path (CSV + SIG) on separate output.

Each row is a rung (spending path). Rungs are OR — satisfy any one rung to spend. Blocks within a rung are joined by + meaning AND — all must pass.

How Dead Man'S Switch (Inheritance) Works
1.
Funding TX creates two outputs. Vout 0 (ALIVE) has SIG + LATCH_SET (state=0) + RECURSE_SAME (max_depth=10). Vout 1 (INHERIT) has CSV (~6 months) + SIG (heir).
2.
The owner proves they're alive by spending vout 0 with a valid signature. RECURSE_SAME enforces the spending output carries byte-identical conditions — the covenant loop.
3.
Each keepalive resets the relative timeout on the INHERIT output. As long as the owner keeps spending within the CSV window, the heir cannot claim.
4.
If the owner fails to sign within the CSV window, the heir satisfies vout 1 by waiting out the lock and providing their signature. The INHERIT path has no recursion — it spends freely.
