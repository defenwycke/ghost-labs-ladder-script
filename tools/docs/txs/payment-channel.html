Payment Channel

Lightning-style payment channel using Ladder Script. The funding transaction creates three outputs: a cooperative close path requiring both parties' signatures, and two unilateral force-close paths each gated by a 24-hour CSV_TIME timeout and a CTV template commitment. This example exercises the cooperative close — both parties sign to settle the channel immediately.

DUAL SIG · COOP CLOSE CSV_TIME · 86400s CTV · TEMPLATE HASH SCHNORR · SCHEME 0x01
Rung Structure

Three rungs across three outputs define the channel protocol. Each output has a single spending condition (one rung per output). The COOP output is the happy path; the FORCE outputs are fallbacks.

The COOP path requires both Alice and Bob's Schnorr signatures — no timelock, no template constraint. The FORCE paths allow unilateral close: one party's signature + a 24-hour CSV_TIME relative timeout + a CTV template hash that constrains where the funds can go (preventing theft during the dispute window).

The evaluator processes both SIG blocks in the COOP rung. Each block verifies the revealed pubkey against its PUBKEY_COMMIT and validates the Schnorr signature. Both must return SATISFIED for the rung to pass.

How Payment Channels Work
1.
The funding transaction creates three outputs: COOP (vout 0) requires both Alice's and Bob's signatures; FORCE_A (vout 1) requires Alice's signature + 24-hour CSV_TIME + CTV template; FORCE_B (vout 2) requires Bob's signature + 24-hour CSV_TIME + CTV template.
2.
Cooperative close (happy path): Both parties agree on the final balance split, co-sign a transaction spending vout 0, and broadcast. No timeout needed. The evaluator checks both SIG blocks — all SATISFIED, rung passes.
3.
Force close (dispute path): If one party is unresponsive, the other can spend their FORCE output after waiting 24 hours (86,400 seconds via CSV_TIME). The CTV template hash constrains the outputs of the force-close transaction, preventing the closer from stealing funds during the timeout window.
4.
This design mirrors Lightning Network commitment transactions but uses Ladder Script's native block types instead of Bitcoin Script opcodes. Each block type is evaluated independently — all blocks in the rung must pass for the spend to be valid.
