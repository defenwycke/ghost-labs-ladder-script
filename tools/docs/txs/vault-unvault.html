Vault with Unvault + Clawback

Bitcoin vault pattern using the VAULT_LOCK block type. A v4 RUNG_TX creates a 2-path output: the hot key can spend after a CSV relative delay (unvault path), while the recovery key can sweep immediately (clawback path). This example funds the vault and then exercises the unvault path by mining 10 blocks to satisfy the CSV delay before spending.

VAULT_LOCK · TYPE 0x0B SCHNORR · SCHEME 0x01

The VAULT_LOCK block is self-contained: it commits both the hot key and recovery key pubkey hashes, plus a numeric delay. The evaluator checks the hot key signature and enforces BIP-68 relative timelock (nSequence) in a single block.

How Vault Spending Works
1.
Funding transaction creates an output with rung_conditions containing two rungs. Rung 0 has a VAULT_LOCK block encoding two PUBKEY_COMMITs (hot key and recovery key) plus a numeric CSV delay of 10 blocks. Rung 1 has a standard SIG block for the recovery key (clawback).
2.
To spend via the unvault path, the spender waits for 10 confirmations, then constructs a transaction with nSequence = 10 on the input. The ladder witness provides the hot key and Schnorr signature.
3.
The evaluator selects rung 0, identifies the VAULT_LOCK block, verifies SHA256(hot_key) matches the committed hash, validates the Schnorr signature against the sighash, and confirms nSequence ≥ 10. Result: SATISFIED.
4.
Alternatively, the recovery key holder can bypass the delay entirely by satisfying rung 1 (CLAWBACK) with a single signature — no CSV wait required. This is the emergency sweep path.
