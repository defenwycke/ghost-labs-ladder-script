Ladder Script Data Flow

Interactive visual guide to the transaction lifecycle, anti-spam architecture, and Merkle mechanics in Ladder Script (BIP-XXXX).

Fund → UTXO → Spend → Verify

1. Fund

Create output
address:raw_address (off-chain)
hash:SHA256(address) → address_hash
conditions:blocks → leaf hashes
pubkeys:folded into leaf (merkle_pub_key)
tree:leaves → Merkle root
output:8 bytes (value only) per output
conditions_root:0xDF || root (33 bytes, one per tx)

2. UTXO

40-byte entry
value:int64 (8 bytes)
root:uint256 (32 bytes)
total:40 bytes, fixed
stored:no conditions, no keys
lifespan:permanent until spent

3. Spend

Witness construction
stack[0]:LadderWitness
stack[1]:MLSCProof
reveals:1 rung + coil + relays
proves:Merkle proof hashes
provides:PUBKEY, SIGNATURE, PREIMAGE
lifespan:ephemeral (prunable)

4. Verify

Consensus validation
step 1:Deserialize witness
step 2:Verify Merkle proof → root
step 3:Merge conditions + witness
step 4:EvalLadder (AND/OR)
step 5:ValidateRungOutputs
step 6:Coil constraints
Key Concepts

merkle_pub_key

Zero writable surface
Public keys are folded into the Merkle leaf hash at fund time. No PUBKEY_COMMIT field exists in conditions. The leaf preimage is:

TaggedHash("LadderLeaf", SerializeRung(rung) || pk1 || pk2 || ... || pkN)

Keys are appended in block order (left to right), count determined by PubkeyCountForBlock(). At spend time, witness provides full keys; verifier recomputes leaf and checks against Merkle root.

address_hash

Coil destination privacy
UNLOCK_TO coils store SHA256(destination_scriptPubKey) instead of the raw address. The raw address is never on-chain in the coil.

address_hash_len: 0 or 32
address_hash: SHA256(raw_spk)

At evaluation time, the verifier computes SHA256 of the spending transaction's output scriptPubKey and compares it to the committed hash.

MLSCVerifiedLeaves

Leaf-centric covenants
Covenant verification uses verified leaf hashes rather than full-conditions comparison. The structure caches:

• conditions_root (committed Merkle root)
• leaf_hashes[] (verified rung + relay + coil leaves)
• n_rungs, n_relays

For mutations: copy leaf array, recompute mutated leaf, rebuild root, compare against output. Only exercised rung and mutation targets are deserialized.

revealed_mutation_targets

Cross-rung mutations
When RECURSE_MODIFIED targets a different rung than the exercised one, the MLSC proof includes the target rung's condition blocks:

[n_mutation_targets: varint]
  [rung_index: uint16]
  [serialised_rung: bytes]

The verifier computes the target leaf hash and confirms it in the Merkle tree. This enables cross-rung mutation without revealing the entire ladder.

Covenant re-encumbrance flow
1
Copy leaves. The verifier copies the input's verified leaf hashes from MLSCVerifiedLeaves.
2
Apply mutation. For the target rung, apply the declared mutation (e.g., RECURSE_MODIFIED delta). Recompute the target leaf hash.
3
Rebuild root. Build a new Merkle root from the modified leaf array using TaggedHash("LadderInternal", min(A,B) || max(A,B)).
4
Compare. Assert that the recomputed root equals the spending transaction's conditions_root (0xDF || root).
Cross-rung mutation via revealed_mutation_targets

Exercised Rung (Rung 0)

Revealed in standard proof
SIG { scheme: SCHNORR }
RECURSE_MODIFIED {
  target_rung: 1
  target_block: 0
  target_field: 0
  delta: -1
}

Target Rung (Rung 1)

Revealed via mutation_targets
CSV { blocks: 52560 }
SIG { scheme: SCHNORR }
COUNTER_DOWN {
  count: 12 → 11
}
The MLSC proof includes revealed_mutation_targets with Rung 1's condition blocks. The verifier: (1) confirms Rung 1's leaf hash matches the Merkle tree, (2) applies the mutation (count 12 → 11), (3) recomputes the leaf hash, (4) rebuilds the root and compares against the output. Rung 1 itself is never exercised — only its structure is revealed for mutation verification.
