// overview

Ladder Script introduces transaction version 4 (RUNG_TX). Spending conditions are declared as named, typed blocks organised into rungs -- a model borrowed from industrial PLC ladder diagrams. Every byte in every block has a declared type with enforced size constraints. Evaluation is deterministic with bounded execution time.

61 block types (59 active + 2 deprecated) across 10 families: signatures, timelocks, hash verification, covenants, recursion, anchors, programmable logic, compound patterns, governance, and legacy Bitcoin transaction types.

// evaluation model

Rungs = OR. Blocks within a rung = AND. First rung where every block returns SATISFIED wins. If no rung passes, the transaction is invalid.

// EvalLadder iterates rungs (OR) // EvalRung iterates blocks (AND) // EvalBlock dispatches to type-specific evaluator Rung 0: SIG(hot_key) AND CSV(144) -> UNLOCK Rung 1: MULTISIG(2-of-3) -> UNLOCK // Rung 0 satisfied? Power flows. Done. // Rung 0 fails? Try Rung 1.

Each rung terminates in a coil: UNLOCK (standard spend), UNLOCK_TO (spend to specific address), or COVENANT (constrain spending tx structure). Relays provide shared condition blocks that multiple rungs reference via relay_refs for cross-rung AND composition.

// anti-spam hardening

Three coordinated defences close all practical data-embedding surfaces:

merkle_pub_key=pubkeys folded into Merkle leaf hash, no writable pubkey slot in conditions
selective_inversion=key-consuming blocks (SIG, MULTISIG, HTLC, ...) cannot be inverted
hash_lock_deprecation=HASH_PREIMAGE/HASH160_PREIMAGE removed; use HTLC, HASH_SIG, or HASH_GUARDED

With MLSC (0xC2), the on-chain output is a fixed 33-byte scriptPubKey regardless of condition complexity. Conditions are never stored on-chain -- only a 32-byte Merkle root appears in the UTXO set.

// post-quantum

Four native PQ signature schemes: FALCON-512, FALCON-1024, Dilithium3, SPHINCS+. A single SCHEME field routes any signature block to classical or PQ verification without changing wire format. With merkle_pub_key, condition size is constant regardless of key size -- a 33-byte Schnorr key and a 1,952-byte Dilithium3 key produce the same Merkle leaf.

COSIGN lets a single PQ-secured UTXO serve as a co-spending guardian for unlimited classical UTXOs -- incremental migration without a flag day.

// typed data model
// Condition fields (stored on-chain) HASH256 // 32 bytes SHA-256 digest HASH160 // 20 bytes RIPEMD160(SHA256) SPEND_INDEX // 4 bytes uint32 LE NUMERIC // 1-4 bytes uint32 LE SCHEME // 1 byte 01=Schnorr 02=ECDSA 10=FALCON-512 ... // Witness fields (revealed at spend time) PUBKEY // 1-2,048 bytes bound by Merkle leaf SIGNATURE // 1-50,000 bytes PREIMAGE // exactly 32 bytes

Static analysis requires only parsing, not execution simulation. Every field can be identified, validated, and displayed without running any code.

// wire format

Two output formats. Inline (0xC1) stores full serialised conditions. MLSC (0xC2) stores only a 32-byte Merkle root -- conditions revealed at spend time in the witness.

Blocks use micro-headers -- a 1-byte lookup table (0x00-0x3C) that encodes block type, inversion flag, and implicit field layout in a single byte. Known blocks skip the 2-byte type code and field metadata entirely. Escape codes (0x80/0x81) fall back to full headers.

// MLSC output: 33-byte scriptPubKey 0xC2 || conditions_root (32 bytes) // Witness structure [n_rungs: varint] [n_blocks: varint] [micro_header: uint8] // 0x00-0x3C lookup [fields...] // implicit layout [coil_type: uint8] [attestation: uint8] [n_relays: varint]
