HASH160_PREIMAGE

RIPEMD160(SHA-256) hash preimage verification. Deprecated as part of anti-spam hardening. Standalone hash preimage blocks with invertible, writable hash fields created a data embedding surface. Use the compound blocks HTLC (hash + signature + timelock) or HASH_SIG (hash + signature) instead.

Ladder Diagram
Fields
FieldData TypeSizeSideDescription
expected_hash HASH160 20 B Conditions Expected RIPEMD160(SHA256) hash value. Node-computed — the user provides PREIMAGE and the node computes this field automatically. Raw hash input is rejected.
witness_hash HASH160 20 B Witness Copy of the expected hash, echoed in the witness for implicit layout matching.
preimage PREIMAGE 32 B Witness Data whose RIPEMD160(SHA256) hash must match the expected value — provided by the user. The node derives HASH160 from this data at creation time.
Wire Format Breakdown

Conditions side (in scriptPubKey):

0x0202 0 1 HASH160 · 20B = 26 bytes

Witness side (in input witness):

0x0202 0 2 HASH160 · 20B PREIMAGE · 32B = 58 bytes
Total (conditions + witness) ~84 bytes

12 bytes smaller than HASH_PREIMAGE due to compact 20-byte hash commitment.

Evaluation Logic
1. PREIMAGE missing → ERROR
2. HASH160 missing → ERROR
3. Compute RIPEMD160(SHA256(PREIMAGE)). If result matches HASH160 → SATISFIED
4. Hash mismatch → UNSATISFIED

Use Cases
Bitcoin Address-Compatible Hashlocks
Hash commitments using the same HASH160 function as Bitcoin addresses. Allows reuse of existing address-generation tooling and familiar 20-byte hash format.
Compact Hash Commitments
When wire size matters, HASH160 saves 12 bytes per commitment compared to HASH256 (20 bytes vs 32 bytes) while maintaining strong preimage resistance.
Legacy Compatibility
Interoperability with systems that already use RIPEMD160(SHA256) for hash commitments, including existing Bitcoin script hash-lock patterns and P2PKH-derived protocols.
