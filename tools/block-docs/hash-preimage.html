HASH_PREIMAGE

SHA-256 hash preimage verification. Deprecated as part of anti-spam hardening. Standalone hash preimage blocks with invertible, writable hash fields created a data embedding surface. Use the compound blocks HTLC (hash + signature + timelock), HASH_SIG (hash + signature), or HASH_GUARDED (non-invertible standalone hash check) instead.

Fields
FieldData TypeSizeSideDescription
expected_hash HASH256 32 B Conditions Expected SHA-256 hash value. Node-computed — the user provides PREIMAGE and the node computes this field automatically. Raw hash input is rejected.
witness_hash HASH256 32 B Witness Copy of the expected hash, echoed in the witness for implicit layout matching.
preimage PREIMAGE 32 B Witness Data whose SHA-256 hash must match the expected value — provided by the user. The node derives HASH256 from this data at creation time.

Evaluation Logic
1. PREIMAGE missing → ERROR
2. HASH256 missing → ERROR
3. Compute SHA256(PREIMAGE). If result matches HASH256 → SATISFIED
4. Hash mismatch → UNSATISFIED

Use Cases
Hash Lock Payments
Conditional payments where funds are released only when the recipient reveals a secret preimage. The sender commits the hash in the output; the recipient proves knowledge of the preimage to spend.
Secret Revelation Proofs
On-chain proof that a specific secret was disclosed. The preimage in the witness serves as irrefutable evidence that the secret holder chose to reveal it.
Commitment Schemes
Two-phase commit-reveal protocols where a party first publishes a hash commitment, then later reveals the committed data by spending with the preimage witness.
