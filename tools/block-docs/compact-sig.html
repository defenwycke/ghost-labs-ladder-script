COMPACT_SIG

Compact rung encoding for single-key signature conditions. Stores SCHEME directly on the rung with no block overhead (public key folded into Merkle leaf via merkle_pub_key). Resolves to a standard SIG block at evaluation time. Primarily used for change outputs and single-key spending.

Wire Format Breakdown

Conditions side (compact encoding):

0x00 0x01 SCHEME · 1B = 3 bytes

The 0x00 byte is the n_blocks sentinel — zero blocks signals compact encoding. The 0x01 byte is the compact type (COMPACT_SIG). SCHEME follows as a raw byte with no field header. The public key is folded into the Merkle leaf via merkle_pub_key.

Witness side (normal SIG block in witness rung):

0x01 SIG μhdr PUBKEY · 32B SIGNATURE · 64B = 100 bytes
Total (compact conditions + witness) 135 bytes

Comparison with normal SIG rung:

EncodingConditionsWitnessTotal
Normal SIG rung35 B100 B135 B
Compact SIG rung35 B100 B135 B

Wire size is identical for the minimal case. The benefit of compact encoding is structural simplification — compact rungs have no block array, no block headers, and no field type tags. This eliminates parsing ambiguity and makes compact rungs trivially identifiable at the wire level. When used with MLSC, compact rungs produce smaller Merkle leaves (35 bytes serialised vs 35 bytes for an equivalent micro-header SIG block — tied in the common case, but compact avoids the implicit-layout machinery).

Evaluation Logic
1. Detect compact rung: IsCompact() returns true. Read compact type = COMPACT_SIG.
2. Resolve: Build an equivalent SIG block with SCHEME field from compact data. Public key resolved via Merkle leaf.
3. During merge, conditions compact data provides SCHEME; public key is folded into Merkle leaf (merkle_pub_key). Witness provides PUBKEY + SIGNATURE.
4. Verify witness PUBKEY against Merkle leaf (merkle_pub_key). If mismatch → UNSATISFIED
5. Verify SIGNATURE against PUBKEY using the committed SCHEME. If invalid → UNSATISFIED
6. Signature valid → SATISFIED

Policy Rules
• Compact type must be known (currently only 0x01 COMPACT_SIG)
• Merkle leaf must contain exactly 1 pubkey (PubkeyCountForBlock = 1)
• SCHEME must be a known scheme identifier
• Compact rungs must not have relay_refs (relay dependency not supported)
Use Cases
Change Outputs
The primary use case. When a wallet constructs a transaction, the change output needs only a single-key signature condition. Compact encoding avoids block-level overhead for this common pattern, keeping the MLSC Merkle leaf minimal.
Simple Single-Key Spending
Any UTXO that requires only a single signature can use COMPACT_SIG. The encoding is semantically identical to a SIG block — it resolves to one at evaluation time — but signals intent as a lightweight spending condition.
MLSC Leaf Optimisation
In MLSC (Merkelised Ladder Script Conditions), each rung becomes a Merkle leaf. Compact rungs serialise to a fixed 35-byte leaf payload with no block-header machinery, making the Merkle proof computation straightforward and the leaf trivially verifiable.
Design Notes
Not a Block Type
COMPACT_SIG is a compact rung encoding, not a new block type in the RungBlockType enum. It uses the n_blocks == 0 sentinel within a rung to signal that the rung carries inline compact data rather than an array of blocks. The 61 active block types (2 deprecated) is unchanged.
Sentinel Design
The wire format uses two distinct zero-sentinels at different levels: n_rungs == 0 signals template inheritance (conditions) or diff witness (witness), while n_blocks == 0 within a rung signals compact encoding. These are unambiguous because they operate at different nesting depths.
Post-MLSC Spam Safety
Before MLSC, compact encodings in conditions raised spam concerns — full conditions were embedded in scriptPubKey. Post-MLSC, the scriptPubKey is always a fixed 33-byte Merkle root (0xC2 + 32B), regardless of conditions complexity. Compact encoding cannot increase the UTXO set footprint.
